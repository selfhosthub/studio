# Self-Host Studio

[![codecov](https://codecov.io/github/selfhosthub/studio-dev/graph/badge.svg?token=BSRFPQC5ZI)](https://github.com/selfhosthub/studio)

> **Community & support:** [SelfHostHub Community](https://www.skool.com/selfhosthub) · [Innovators (Plus)](https://www.skool.com/selfhostinnovators)

A self-hosted workflow automation platform for media workflows, templates, and distributed job processing.

## Quick Start

The fastest way to manage a self-hosted Studio instance ([Docker](https://docs.docker.com/get-docker/) + Compose v2 required):

```bash
# 1. Install uv (one-time)
curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc           # macOS default shell is zsh - use ~/.zshrc instead

# 2. Install studio-console
uv tool install https://github.com/selfhosthub/studio-console/releases/download/v1.0.0/studio_console-1.0.0-py3-none-any.whl

# 3. Run
studio-console
```

On first run with no `~/.studio/.env`, the setup wizard launches automatically. It validates Docker, generates secrets, prompts for an admin email and domain, writes `~/.studio/.env`, pulls images, and starts `postgres + api + ui`.

Access (defaults):
- **API**: http://localhost:8000
- **UI**: http://localhost:3000
- **API Docs**: http://localhost:8000/docs

Credentials are set during the wizard; use `studio-console reset-password` if you need to reset them later.

## Deployment Shapes

Studio runs on one host or many. Pick the shape that matches your hardware:

| Shape | Containers | When |
|-------|-----------|------|
| **Standard** | 3-8 (postgres + api + ui + optional workers) | Solo operator, one VPS or home server |
| **Core** | 2 (postgres + studio-core image) | Fewer containers on a single host |
| **Split** | 4-8 across hosts | Scaling out API, UI, and workers separately |
| **Distributed workers** | API host + worker host(s) | GPU workers on a different machine from the API |
| **RunPod / cloud GPU** | API anywhere + RunPod GPU pod | On-demand GPU bursts |
| **Full** | 1 | Constrained environments, evaluation |

Decision tree, resource requirements, and per-shape compose commands are in [production/deployment-matrix.md](production/deployment-matrix.md).

## Daily Operations

```bash
docker compose up -d          # start core (postgres + api + ui)
docker compose --profile workers up -d   # add all workers
docker compose logs -f api    # follow API logs
docker compose down           # stop (data preserved)
```

GPU workers use the GPU compose override:

```bash
docker compose -f docker-compose.yml -f workers/docker-compose.gpu.yml up -d
```

For management tasks (reset admin password, rotate secrets, edit env), use `studio-console`.

## Configuration

On first boot the bootstrap script reads `/workspace/.env` (mapped to `~/.studio/.env` on the host). If it doesn't exist, the production template is copied and Studio refuses to start until required values are set.

The setup wizard collects:
- What runs on this machine (API + UI, API + UI + Workers, or Workers only)
- Public URL where Studio will be reached
- Database connection string
- Security keys (auto-generated by default)
- Admin credentials for first login

Once `~/.studio/.env` exists it is never overwritten. To edit values later:

```bash
studio-console config set KEY VALUE   # set a single value
studio-console wizard                 # re-run the full setup wizard
```

For the full reference of every env var Studio reads, see [production/env-vars.md](production/env-vars.md).

## Workers on a Separate Machine

If workers run on a different host from the API, run `studio-console` on the worker host and select Workers-only when the wizard asks what to deploy. You'll need the `SHS_WORKER_SHARED_SECRET` from the API host - copy it byte-for-byte. See [production/deployment-matrix.md](production/deployment-matrix.md#distributed-workers) for the topology.

## RunPod

Use the Full or distributed-worker topology. Mount a network volume at `/workspace` - without it, `/workspace/.env` is lost when the pod is destroyed.

The Core and Full images run multiple processes under supervisord, whose web dashboard is exposed on port `9001`. RunPod auto-publishes that port through its TCP proxy, so basic-auth is required: set `SHS_SUPERVISOR_USER` and `SHS_SUPERVISOR_PASSWORD` (generate with `openssl rand -base64 24`) before starting the pod. Supervisord refuses to start if either is unset.

Full RunPod walkthrough: [production/vps-runpod.md](production/vps-runpod.md).

## Project Layout

```
studio/
├── api/              # FastAPI backend (Python 3.12)
├── ui/               # Next.js frontend
├── workers/          # Job workers (Python 3.11)
├── contracts/        # Shared Pydantic models (API ↔ workers)
├── deploy/           # .env.example template + nginx config
├── docs/             # Operator and infrastructure docs
├── docker-compose.yml             # Production services
├── workers/docker-compose.gpu.yml # GPU worker override
└── Dockerfile, Dockerfile.full
```

Provider packages, content catalogs, and marketplace packages are deployed separately - the wizard pulls them at first boot.

## Tech Stack

- **API**: Python 3.12, FastAPI, PostgreSQL 18 (pgvector), Redis, SQLAlchemy (async)
- **UI**: Next.js, TypeScript, Tailwind CSS, React Query
- **Workers**: Python 3.11, Pydantic, httpx
- **Auth**: JWT with role-based access control
- **Real-time**: WebSockets via Redis pub/sub
